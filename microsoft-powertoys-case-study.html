Case Study Capstone

Keystroke Overlay for Microsoft PowerToys

A new utility for Microsoft PowerToys that visualizes keyboard inputs in real time, helping educators, presenters, and developers show their keystrokes during demos and recordings.

My role Product Designer, contributed to development
Tools Figma · C# · WinUI3 · XAML
Client Microsoft (via RPI Capstone)

Context

A real feature for a real product, with a real client

This was an RPI capstone project in collaboration with Microsoft. Our team of five built a new Screencast Mode utility for Microsoft PowerToys, the open-source productivity suite used by over 5 million Windows users. We worked directly with Kayla Cinamon, an RPI alumna and Product Manager at Microsoft, who gave weekly feedback on both design and technical decisions.

The project ended with a production-ready pull request submitted to the PowerToys GitHub repository, following Microsoft's official contribution guidelines. It was not a simulation or a class exercise. It was a real feature intended for real users.

The problem

Viewers cannot see what keys the presenter is pressing

When developers, educators, and content creators record tutorials or give live demos, there is no built-in way on Windows for viewers to see which keys are being pressed. Third-party tools exist but feel out of place and require separate installs. There was an open GitHub issue on the PowerToys repository requesting this feature.

The goal was to build something that felt native to Windows and PowerToys, lightweight enough to stay out of the way, and configurable enough to work across different use cases.

My role

Design lead, with a hand in the code

I owned the product design for Screencast Mode. That meant designing the overlay UI and the settings panel in Figma, making decisions about layout, visual hierarchy, and the appearance customization options, and ensuring the designs matched PowerToys' Fluent Design language.

I also contributed to the C# implementation, translating the Figma designs into XAML and WinUI3 components within the PowerToys codebase. The design and engineering were tightly coupled throughout, and working across both gave me a clearer sense of what was technically feasible at each stage.

Working with Kayla changed how I think about design feedback. She pushed us to simplify at every turn, to cut things that felt nice to have but didn't serve the core use case. The final design is more focused because of that.

Design constraints

Designing inside an existing system

This was not a greenfield project. PowerToys has an established visual language, settings structure, and user expectations. Every design decision had to work within that context.

  • Fluent Design compliance. The settings panel had to match the existing PowerToys UI exactly, using the same layout patterns, typography, and controls as every other utility in the suite.
  • Performance limits. The overlay had to stay under 3% CPU usage and 60 MB memory, and display keystrokes within 50ms. That ruled out anything visually heavy.
  • Minimal footprint. The overlay needed to be visible without being distracting. It should never obscure the content the presenter is trying to show.
  • Scope discipline. Microsoft pushed us to prioritize. Features like multi-monitor support, drag-and-drop positioning, and gesture visualization were deliberately deferred to keep the core delivery clean and on time.

Process

Weekly sprints, weekly client feedback

The team ran one-week sprints with a standing meeting with Kayla each week. Design decisions went to her for review before we committed to implementation. This kept scope tight and meant we were never building something the client hadn't signed off on.

The designs

Two components, two distinct design problems

Screencast Mode has two parts: the overlay that appears during use, and the settings panel that lives inside the PowerToys dashboard. Each had different design goals.

The settings panel

The settings panel needed to feel like it had always been part of PowerToys. I followed the existing layout patterns precisely: a master enable toggle at the top, an activation section for the hotkey, and an appearance section for customization. Controls include overlay position (a visual 3x2 grid selector), auto-hide delay, background color, font color, and font size.

Settings panel wireframe: master toggle, activation hotkey, and appearance controls including position selector, colors, and font size

The overlay

The overlay had one job: show what key was just pressed, clearly and unobtrusively. I designed it as a small pill in the corner of the screen displaying the keystroke combination in real time. The position is configurable to any of six screen corners. It auto-hides after a set delay so it does not linger when the presenter stops typing.

Overlay wireframe: a small pill in the top-right corner showing the active keystroke combination, designed to be visible without obscuring the presenter's content

Key design decisions

What I decided and why

Position selector as a visual grid, not a dropdown

For overlay position, I designed a 3x2 grid of clickable cells representing the six screen corners rather than a text dropdown. This makes the spatial relationship immediately obvious, a dropdown listing "Top Left, Top Center, Top Right" requires the user to mentally map words to positions. The grid does it visually at a glance.

Auto-hide over a manual close

The overlay fades automatically after a configurable delay rather than requiring a manual dismiss. During a presentation or recording, the presenter's hands are busy. An overlay that cleans itself up after 3 seconds is far less disruptive than one that stays until explicitly closed.

Cutting multi-monitor support from v1

We scoped this out early on Kayla's advice. Supporting multiple monitors would have required significantly more engineering complexity and testing time. The core use case, a single presenter on a single screen, was well served without it. Deferring it kept the feature clean and deliverable on schedule.

Matching PowerToys exactly, not approximating it

Every control, spacing value, and typography choice in the settings panel came from the existing PowerToys component library. I referenced the live app and open-source code throughout design to make sure nothing felt out of place. A utility that looks slightly off erodes trust in the whole suite.

Reflection

What designing for an existing product teaches you

Designing for PowerToys was different from any personal project I had done. There was no blank canvas. Every decision had to justify itself within a system that already had rules, users, and expectations. That constraint was clarifying rather than limiting it forced precision over preference.

Getting weekly feedback from an actual Microsoft PM also changed how I think about iteration. Kayla's consistent push toward simplicity, cutting the multi-monitor feature, keeping the overlay minimal, resisting the urge to add customization options that would complicate the settings panel, made the final product better. Design by subtraction is harder than design by addition, and this project was good practice for it.
