Case Study Spring 2026

CareCompass

A privacy-preserving care coordination platform that lets independent clinics close preventive care gaps across a patient network without sharing a single patient record.

My role Dashboard design, UI system, demo flow
Tools Figma · JS · Flask · Python

The problem

Patients fall through the cracks and clinics can’t see why

James is 34 and manages bipolar disorder across two clinics a psychiatrist at Clinic A for medication, and group therapy at Clinic B. In early March he missed two consecutive follow-ups at Clinic A. The system flags him as a care gap. What Clinic A doesn’t know is that James had a crisis episode in February and has been attending intensive outpatient sessions at Clinic B three times a week since then. He isn’t disengaged. He’s more engaged than he’s been in years.

Unnecessary outreach to a patient in active psychiatric treatment can disrupt the therapeutic relationship. A false care gap flag can follow a patient across systems. And the inverse failure a patient who genuinely has gone silent can be fatal.

The straightforward fix, pooling records across clinics, is blocked by HIPAA. CareCompass solves the visibility problem without moving any patient data.

System architecture

How it works without sharing data

CareCompass has three layers, each addressing a distinct constraint. Each clinic runs its own local node. Patient identifiers are hashed with SHA-256 and a shared secret salt the same patient produces the same anonymous token at any clinic. Cross-clinic queries send only the token and receive only a boolean. A federated learning model trains locally at each clinic and shares only weight updates, not records.

System architecture: decentralized clinic nodes, identity hashing layer, and federated learning components

The hashing pipeline raw PII never leaves the local node. Only the 64-character token crosses clinic boundaries.

Design constraints

Privacy-first, from the first sketch

These weren’t aesthetic constraints. They were hard technical and ethical requirements that shaped every UI decision.

🔒

No data crosses boundaries

The response to any cross-clinic query is a single boolean. The UI had to surface actionable information from an intentionally minimal signal.

👥

Non-technical primary users

Clinic receptionists and care coordinators, not engineers. Complex federated learning outputs had to become plain, immediate language.

⚠

High-stakes decisions

A missed patient in a psychiatric network carries real clinical risk. The UI couldn’t let users misread a degraded result as a definitive negative.

📄

No third-party frameworks

Given the sensitivity of psychiatric data, the codebase had to stay small and easy to audit. Every component was built from scratch.

Key design decisions

The choices that shaped the product

Three views, one mental model

The dashboard is organized into three views: overview (population-level gap detection), patient lookup (individual cross-clinic reconciliation), and network panel (infrastructure status). Each view has one job and one primary action. In a clinical setting, cognitive load is a safety issue.

Three patient states, immediately scannable

The overview surfaces patients into three categories. The color and label system had to make the right action obvious at a glance, without requiring the user to understand the underlying system.

Missing everywhere

No recent visit anywhere in the network. Immediate outreach needed.

Seen elsewhere

Overdue locally, but in active care at a peer clinic. No outreach needed.

Local only

Status unknown to the network. Use the federated risk score to guide next steps.

The overview dashboard three patient states, ML risk scores, and one-click outreach actions

Hashing happens client-side

When a receptionist looks up a patient, their name and date of birth are hashed in the browser using the Web Crypto API before anything is sent over the network. Only the SHA-256 token leaves the local machine. I designed the lookup form to surface this explicitly a visible “Privacy Protected” notice explains that no identifying information ever leaves the device. In a psychiatric context, a patient’s presence at a clinic is itself sensitive information.

Patient lookup hashing happens in the browser, the “Privacy Protected” notice makes this visible to the user

Degraded state is visible, not silent

If a clinic node goes offline during a cross-clinic query, the API returns a partial response. I designed the network panel to flag exactly which node was unreachable, and the lookup result to mark itself as potentially incomplete. A false negative in this system concluding a patient isn’t in care because one node didn’t respond carries real risk. The UI treats silence as ambiguity, not absence.

Network panel Riverside Clinic shows as Degraded (orange), so any lookup result is flagged as potentially incomplete

Reflection

What I learned

CareCompass pushed me to design for two audiences simultaneously clinical staff who need immediate clarity, and engineers who need a system they can build and audit. The privacy constraint wasn’t a limitation I designed around; it was a design input. The fact that the system intentionally knows as little as possible shaped what the UI could and couldn’t say, and I had to make that restraint feel intentional rather than broken to the user.

Making the hashing client-side was a technical choice. Surfacing it visibly to the user, with plain language about what leaves their machine and what doesn’t, was a design choice. Those aren’t the same thing and getting both right required design and engineering to be in conversation the whole way through.
